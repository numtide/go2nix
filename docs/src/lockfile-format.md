# Lockfile Format

go2nix uses TOML lockfiles to pin module hashes and (optionally) the package
dependency graph. Each builder mode uses a different lockfile variant.

| Mode | Command | Format | Sections |
|------|---------|--------|----------|
| Vendor | `go2nix generate --mode=vendor` | v1 (gomod2nix) | `[mod."key"]` attrsets |
| DAG | `go2nix generate --mode=dag` (default) | v2 | `[mod]` + `[replace]` |
| Dynamic | `go2nix generate --mode=dynamic` | v2 | `[mod]` + `[replace]` |

## Vendor format (v1 / gomod2nix)

Generated by `go2nix generate --mode=vendor`. This follows the
[gomod2nix](https://github.com/nix-community/gomod2nix) lockfile format —
existing gomod2nix lockfiles are compatible with go2nix vendor mode.

```toml
# go2nix lockfile (gomod2nix format). Generated by go2nix. Do not edit.

[mod."github.com/foo/bar@v1.2.3"]
version = "v1.2.3"
hash = "sha256-abc..."
replaced = "github.com/fork/bar"
```

Each module is an attrset with `version`, `hash`, and an optional `replaced`
field. Replacements are folded into each module entry rather than a separate
section. There is no `[pkg]` section — vendor mode doesn't need the package
graph.

go2nix internally generates a v2 lockfile and converts it to v1.

## v2 format (DAG and Dynamic)

Generated by `go2nix generate` (default, DAG) or `go2nix generate --mode=dynamic`.
Both DAG and dynamic modes use the same lockfile format.

```toml
# go2nix lockfile v2. Generated by go2nix. Do not edit.

[mod]
"github.com/foo/bar@v1.2.3" = "sha256-abc..."
"golang.org/x/sys@v0.20.0" = "sha256-def..."

[replace]
"github.com/foo/bar@v1.2.3" = "github.com/fork/bar"
```

### Sections

**`[mod]`** — Module hashes. Each key is a composite `"path@version"` string,
each value is a `sha256-...` SRI hash of the module's GOMODCACHE directory.

**`[replace]`** — Module replacements (from `go.mod` `replace` directives).
Maps the original composite key to the replacement module path. Only
remote replacements are recorded; local `replace` directives (filesystem
paths) are not included.

### Composite keys

Module keys use `"path@version"` format (e.g., `"golang.org/x/sys@v0.20.0"`).
This keeps each module uniquely identified and avoids collisions across
versions.

### Package graph resolution

The lockfile does not contain a package dependency graph. Instead:

- **DAG mode** discovers the package graph at eval time via the go-nix-plugin
  (`builtins.resolveGoPackages`), which runs `go list` against the source tree.
- **Dynamic mode** discovers the package graph at build time via `go list
  -json -deps` inside a recursive-nix wrapper.

Only module NAR hashes are required in the lockfile (for fixed-output
derivations that fetch each module).

## Format comparison

| Aspect | Vendor (v1 / gomod2nix) | DAG / Dynamic (v2) |
|--------|-------------|----------|
| Mod values | Attrset (`version`, `hash`, `replaced`) | Plain hash string |
| `[replace]` section | No (folded into `replaced` field) | Yes |
| Typical size | Small | Small |
| Regenerate when | `go.mod` changes | `go.mod` changes |

## Monorepo support

When `go2nix generate` is given multiple directories, all modules are merged
into a single lockfile. Modules from different `go.mod` files coexist without
conflict since each is uniquely keyed by `"path@version"`.

## Staleness detection

Detection varies by mode since each uses different lockfile sections:

| When | What | Applies to | How |
|------|------|-----------|-----|
| Generation | MVS consistency | All modes | `go list -json -deps` resolves actual versions |
| Nix eval | Package graph | DAG only | `builtins.resolveGoPackages` runs `go list` at eval time |
| Build time | Lockfile consistency | DAG, dynamic | `go2nix check --lockfile` validates against `go.mod` |
| Build time | Tidiness | Vendor only | `go mod graph` against GOMODCACHE from vendor tree |

In DAG mode, missing or mismatched packages are caught at eval time when
`builtins.resolveGoPackages` runs `go list`. In dynamic mode, they are caught
at build time when `go list` runs inside the recursive-nix wrapper.

Run `go2nix check <dir>` or `go2nix check --lockfile <path> <dir>` to verify
a lockfile without building.
